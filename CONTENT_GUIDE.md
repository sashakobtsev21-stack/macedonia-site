# CONTENT_GUIDE.md — как добавить SEO-статью на North Macedonia Guidebook

Рабочая инструкция для регулярного написания статей. **Источники правды по продукту/правилам:** `SPEC.md` → `CLAUDE.md` → этот файл. Если что-то противоречит — приоритет у SPEC/CLAUDE. Доказательная база фактов о стране — `docs/MACEDONIA-ANALYSIS-2026-06-21.md` (всё веб-проверено), стратегические решения — `docs/STRATEGY.md`.

> TL;DR: статья = Markdown-файл в `src/content/articles/en/<slug>.md` (Astro Content Collections). Сайт **одноязычный, только en (на корне `/`)**: статья отдаётся на корне `/<category>/<slug>/`. Публикуется **только en-версия**. Перед публикацией — `npm run qa` → должно быть **GO**.

---

## 0. Для владельца: как это работает (простыми словами)

**Главное: статьи пишешь НЕ ты.** Их пишу я: черновик EN (сайт одноязычный, только en) → фото → **взаимная перелинковка** (новая статья ссылается на соседей+хаб — это гейт; И опорная/соседи кластера ссылаются на неё — мой ручной шаг, гейтом не ловится) → сверка фактов (`fact-checker`) → финальный приёмщик (`content-editor`) → `npm run qa` (= GO) → коммит + публикация. Твоя роль короткая:

1. **Дать факты, которых нет в открытых источниках** — особенно местное по Охриду/Скопье/Битоле (любимые места, цены по чеку, что реально стоит/не стоит, что закрылось). Без этого я пишу только на **проверяемых фактах** из надёжных источников — без выдуманного «я там был». Владелец **не живёт** в Македонии — моат держится на свежести, структуре и оригинальном контенте (STRATEGY §5).
2. **Дать свои фото** (по желанию) — они дают метку «Проверено» и сильнее в Google, чем сток. Нет своих → беру бесплатные CC.
3. **Сказать «ок»** на публикацию.

**Как заказать статью:** просто напиши «пиши про &lt;тему&gt;» — я беру тему из плана, собираю факты, пишу en-версию, прогоняю гейты и публикую. Или кинь мне факт/место — я превращу в контент.

**По какому плану:** очередь и приоритеты — `docs/CONTENT_PLAN.md` (когда будет создан: §0 — стратегия, дальше кластеры/очередь). Пока ориентир — кластерный контент-план в `docs/MACEDONIA-ANALYSIS-2026-06-21.md` §6 и приоритеты `docs/STRATEGY.md` §4.

**Приоритет кластеров (STRATEGY §4 — релокация/номады в приоритете):**
1. **Планирование** (`/planning/`): виза, валюта-денар, когда ехать, как добраться — верх воронки, донор трафика.
2. **MONEY-кластер:** `/car-rental/` · `/insurance/` · eSIM — транзакционные прямые `/go/`-связки.
3. **Релокация/Номады** (`/relocation/`, EN-first): cost of living, регистрация компании (DOO/DOOEL), **плоский налог 10%**, ВНЖ — приоритетный маржинальный рынок.
4. **Достопримечательности / Города / Маршруты** — упор на **озеро Охрид** (ЮНЕСКО) и природу (нацпарки Маврово, Галичица, Пелистер); лёгкий вход — Битола и вторичные точки (Крушево, Маврово, каньон Матка, Преспа).
5. **Еда.** 6. **Развлечения.**

**Правило фактов (железное, §15):** цены/адреса/часы — только из надёжного источника + «уточняйте» + дата; нет данных → не выдумываем. Для YMYL-тем (визы/налоги/ВНЖ) — только официальные первоисточники (§14). Это защита от бана Google.

## 1. Как технически устроены статьи

- **Движок:** Astro 6 + **Content Collections**. Статьи — это **Markdown** (`.md`/`.mdx`) с YAML-frontmatter. Отдельные `.astro`-файлы под каждую статью НЕ создаются — рендерит общий шаблон `src/layouts/ArticlePage.astro` через динамический маршрут.
- **Где лежат файлы:**
  ```
  src/content/articles/en/<slug>.md   ← английская версия (сайт одноязычный, только en)
  ```
  (Маршруты — отдельная коллекция `src/content/routes/`; «Где поесть» — `src/content/restaurants/`; услуги — `src/content/services/`; города-карточки — `src/content/cities/`. Эта инструкция — про **статьи** в `articles/`. Папок `/ru/`, `/uk/` нет.)
- **Как файл попадает в нужный раздел:** разделом управляет поле **`category`** во frontmatter, а НЕ папка. URL строится как `/<category>/<slug>/` (без языкового префикса — сайт одноязычный). Файл `articles/en/abc.md` с `category: 'cities'` → страница `/cities/abc/` и автоматически попадает в хаб `/cities/` (хаб сам подтягивает статьи своей категории — ручных списков нет).
- **Допустимые `category`** (enum в `src/content.config.ts`, английские слаги):
  `attractions` · `cities` · `food` · `entertainment` · `routes` · `transport` · `car-rental` · `relocation` · `insurance` · `news` · `planning`.
  - Для обычных SEO-статей чаще всего: **cities, attractions, transport, car-rental, relocation, planning, news, entertainment**.
  - Нюансы: `routes` — это отдельная коллекция `routes/` (не `articles/`); `food` — директория мест (`restaurants/`), а не лонгрид; `insurance` — раздел с кастомным хабом, но статьи категории `insurance` в `articles/` тоже работают (`/insurance/<slug>/`); `relocation` имеет под-раздел услуг (`/relocation/services/`).

## 2. Frontmatter статьи (поля)

Полная схема — `src/content.config.ts` (`articleBase`). Главное:

| Поле | Обяз.? | Что это |
|---|---|---|
| `title` | да | Заголовок = **H1** на странице. **≤ 60 символов** (гейт `check-parity`). Должен содержать целевой запрос (см. §8). |
| `description` | да | Meta description. **≤ 155 символов** (валидируется схемой). 1–2 предложения с запросом. |
| `slug` | да | Латиница-цифры-дефис. **Стабилен навсегда** (URL не меняем — правило 3). |
| `lang` | да | `'en'` (сайт одноязычный) — должен совпадать с папкой. |
| `category` | да | Один из enum выше → определяет раздел и URL. |
| `publishedAt` | да | Дата публикации `YYYY-MM-DD`. |
| `updatedAt` | да | Дата обновления `YYYY-MM-DD` (видна на странице — «Обновлено · …»). Поднимай при правках фактов. |
| `yearInTitle` | нет | `true` → подставит текущий год в конец title/H1/schema на сборке (для «денежных» гайдов P1-2). |
| `cover` | да* | Обложка: `src` (путь к webp) + `alt`. *Фото обязательны (правило проекта); гейт `check-photos` валит сборку без существующего cover ≤200 КБ. |
| `coverCredit` | для CC | Атрибуция обложки «Фото: Автор / источник, лицензия» — обязательна для CC/чужих фото (§18). |
| `gallery` | нет | Массив `{src, alt, credit?}` — мини-галерея с лайтбоксом. |
| `affiliate` | нет | 1–3 партнёрских бокса (см. §7). По умолчанию `[]`. |
| `sources` | нет | Массив доменов/ссылок-первоисточников фактов (§14). |
| `verifiedAt` | нет | Дата реальной сверки места → бейдж «Проверено · …». Без сверки — НЕ ставить (будет «Обновлено», правило 4). |
| `geo` | нет | `{ coord: [lat, lng], address? }` — для статей о местах (даёт карту + schema). |
| `accessFrom` | нет | «Как доехать» из Скопье/Охрида/Битолы (`{km?, duration?, note?}` под ключами `skopje`/`ohrid`/`bitola`). |
| `visit` | нет | «Вход и часы»: `{price?, hours?, note?, checkedAt?}`. Цены/часы НЕ выдумывать — только источник + дата + «уточняйте». |
| `featuredOrder` | нет | Меньше число = выше в хабе раздела. |
| `attractionType` / `region` | нет | Только для `attractions` (фильтры каталога). Слаги — в `content.config.ts`. |
| `razvlType` | нет | Только для `entertainment` (`nightlife`/`events`/`active`/`casino`/`places`). |
| `tags` | нет | Свободные теги (показ в смежных разделах). |
| `hotelWidget` | нет | `true` → виджет поиска отелей (для городов с высоким спросом на жильё). |
| `qualityMark` | нет | Знак качества (медаль) — ТОЛЬКО по запросу владельца (§13). По умолчанию `false`. |
| `website` / `instagram` | нет | Для статей-мест (клуб/казино и т.п.) — кнопки «Сайт»/«Instagram». |
| `draft` | да на старте | `true` = скелет, в сборку НЕ попадает. Снять (`false`) только когда статья готова и гейты зелёные. |

**Примеры** (наполняются в контент-фазе — на R1 коллекции пустые): после создания первых статей смотри их целиком как референс — справочную статью со страховкой (`affiliate` + `sources`), статью о месте (`geo`, `gallery`, инлайн-фигуры, `accessFrom`), статью-город (`featuredOrder`, `hotelWidget`, `visit`).

## 3. Язык (только EN) и hreflang

- **Сайт одноязычный: только en на корне `/`.** Папок `/ru/`, `/uk/` нет (ru/uk удалены 2026-06-22). `LANGS = ['en']`, `DEFAULT_LANG = 'en'` (`src/i18n/types.ts`).
- **Чтобы статья попала на корень `/`:** файл в `src/content/articles/en/<slug>.md`, `lang: 'en'`, `category: '<…>'`. URL получится `/<category>/<slug>/` автоматически (без языкового префикса).
- **🔴 Главное правило (CLAUDE правило 1):** статья создаётся **только в en** (`src/content/<коллекция>/en/<slug>.md`). Никаких ru/uk-версий и переводов. Гейт проверяет только en-файлы (см. §9).
- **hreflang — автоматический.** `BaseLayout` сам проставляет **self `en` + `x-default`** (оба указывают на canonical URL страницы — сайт одноязычный). От тебя ничего не требуется. Переключателя языка на сайте нет.

## 4. Пошагово: добавить статью (EN-first)

1. **Бриф/факты.** Собери факты заранее (цены, адреса, часы — из надёжных источников). Заводи `briefs/<slug>.md` — ключевой запрос, вопросы читателя, факты владельца/первоисточника, цены, фото. Нет факта → ставь `<!-- TODO: факт от владельца -->` и не выдумывай (правило 4). Для YMYL-тем (визы/налоги/ВНЖ) собирай факты только из официальных первоисточников и фиксируй их в `briefs/_SOURCES-YMYL.md`.
2. **Скелет (рекомендуется).** Из корня проекта:
   ```
   npm run new -- article <slug> --category <category> --title "Заголовок ≤60 символов"
   ```
   Создаст **только** `articles/en/<slug>.md` (схема-корректный frontmatter, `draft: true`, DoD-чеклист в комментарии) + папку фото `public/images/<slug>/`. Существующие файлы не перезатирает.
3. **Текст EN.** Заполни `articles/en/<slug>.md`:
   - `description` (≤155), `publishedAt`/`updatedAt`.
   - **Тело:** начинается с лид-абзаца (это первый абзац, без `#`), затем секции `## H2` / `### H3`. **`#`/H1 в теле НЕ ставить** — H1 рендерится из `title`.
   - Соблюдай SEO-требования из §8.
4. **Фото** (§6) — cover обязателен + ≥3 инлайн-фигуры ИЛИ ≥6 gallery.
5. **Внутренние ссылки** (§7) — минимум 2 на смежные статьи + ссылка на хаб раздела. Для балканских/кросс-граничных тем — кросс-линки на сайты сети (STRATEGY §6).
6. **Сними `draft`** → `draft: false` в en-версии.
7. **Гейты** (§9) — `npm run qa` → **GO**. Для YMYL-тем — обязательная сверка фактов по первоисточникам (`fact-checker`), затем `content-editor`.
8. **Коммит** en-версии → push в `main` (Cloudflare соберёт и задеплоит).

## 6. Фотографии

- **Уникальные на статью (правило владельца):** у каждой статьи СВОИ фото — **НЕ переиспользовать кадры из других статей** (`/images/<другой-slug>...`). Под новую статью искать новые кадры через пайплайн и класть в свою папку. Проверять, что НА фото (не выдавать чужую страну/валюту за македонскую — Северная Македония, денар, кириллица).
- **Где лежат:** `public/images/<slug>/` (создаётся скаффолдером). Формат — **`.webp`**, каждое **≤ 200 КБ** (перф-бюджет §15), с заданными `width`/`height` (нулевой CLS).
- **Откуда брать:** только свои фото или **CC/PD** (Wikimedia / Openverse / Unsplash). Скрипты-помощники: `scripts/commons-candidates.mjs`, `scripts/openverse-candidates.mjs`, `scripts/unsplash-candidates.mjs` (поиск кандидатов) → `scripts/build-gallery.mjs` / `build-unsplash.mjs` (скачать + сжать в webp ≤200 КБ). Атрибуцию (автор+лицензия) бери **дословно из источника**, не из имени файла.
- **Только лучшие кадры (правило владельца).** Отбор глазами; слабые/мусорные кадры не публиковать. Цель по объёму: Скопье/Охрид ~20 фото, города ~12–15 «по качеству».
- **Обложка (frontmatter):**
  ```yaml
  cover:
    src: '/images/<slug>/cover.webp'
    alt: 'Осмысленное описание кадра'
  coverCredit: 'Фото: Имя Автора / Wikimedia Commons, CC BY-SA 4.0'
  ```
- **Инлайн-фото в тексте** (ключевые кадры — прямо в теле, с подписью). Точная разметка (raw HTML в .md):
  ```html
  <figure class="figure">
  <img src="/images/<slug>/g1.webp" alt="Что на фото" width="1280" height="853" loading="lazy" decoding="async" />
  <figcaption>Подпись — что на фото. <span class="figure__credit">Фото: Автор / источник, лицензия</span></figcaption>
  </figure>
  ```
  `width`/`height` = реальные пиксели файла (иначе перекос/CLS).
- **Мини-галерея внизу** (необязательные кадры) — через frontmatter:
  ```yaml
  gallery:
    - src: '/images/<slug>/g2.webp'
      alt: '…'
      credit: 'Фото: …'
  ```
  Каждое фото используется **один раз** (либо инлайн, либо в галерее).
- **`alt`** — осмысленный, по делу (не «фото1»); описывает содержимое кадра.
- **После добавления обложек** прогони `npm run build:covers` — сгенерит адаптивные варианты (`srcset`) и обновит манифест `src/data/cover-variants.json` (мобайл грузит меньший размер). Варианты коммитятся.

## 7. Внутренние ссылки

- **Формат:** обычная Markdown-ссылка с завершающим слэшем, **без языкового префикса** (сайт одноязычный, все ссылки — на корневой путь):
  ```markdown
  Подробнее — в путеводителе по [Охриду](/cities/ohrid/) и в гайде по [аренде авто](/car-rental/).
  ```
- **Минимум 2** внутренние ссылки в теле на смежные статьи — это **гейт** `check-interlinks` (DoD §8.1). Партнёрские `/go/`-ссылки и якоря в счёт не идут.
- Ставь ссылку и **на хаб раздела** (`/<category>/`), и желательно встречную ссылку из смежной статьи (двунаправленный кластер).
- **Кросс-линки сети (STRATEGY §6).** Сайт — часть балканской сети (Албания + Черногория + Хорватия + Македония). Для балканских/кросс-граничных тем предусматривай кросс-линки на сайты сети и общие мульти-страновые роуд-трип-страницы: «Lake Ohrid — two-country guide» (Охрид + Поградец, AL ~40 км/1 ч), «Balkan Grand Loop» (MK→AL→ME→HR), «Ohrid–Tirana–Adriatic».
- **Партнёрские ссылки — ТОЛЬКО через `/go/`** (правило 2), и только во frontmatter `affiliate` (рендерится как AffiliateBox), не прямыми URL в тексте:
  ```yaml
  affiliate:
    - partner: 'booking'          # ключ из src/data/partners.json
      label: 'Где остановиться'
      position: 'body-end'
  ```
  Ключи партнёров — из `src/data/partners.json`. Покрытие Северной Македонии подтверждено анализом (§16): **Booking** (1 817 объектов Скопье / 1 467 Охрид) · **GetYourGuide / Viator** (туры Скопье/Охрид) · **DiscoverCars** (SKP ~24 варианта) · **LocalRent** (~63 авто, RU-интерфейс) · **Airalo** (eSIM, оператор A1) · **Aviasales** (авиапоиск, Travelpayouts) · **SafetyWing** (страховка/номады, EN, рекуррент) · **Wise** (переводы, релокация EN). Многие связки подключаются через Travelpayouts. Партнёр привязывается к теме (отели — к городам, аренда — к транспорту, страховка/Wise — к релокации/планированию).

## 8. SEO-требования к статье (соблюдаем ВСЕГДА)

1. **H1 = целевой запрос.** Поле `title` (= H1) точно содержит целевой поисковый запрос. Ограничение: **≤ 60 символов** (гейт). Один H1 на странице (он из `title`; в теле `#` не использовать).
2. **Первый абзац — прямой ответ на запрос в 2–3 предложениях.** Лид сразу даёт суть, без «разгона».
3. **Подзаголовки `## H2` / `### H3` со смежными запросами** — структура под вопросы читателя (что/где/как/сколько/когда). Логичная иерархия без скачков уровней.
4. **Таблицы** для цен / расстояний / сравнений — обычные Markdown-таблицы (`| … | … |`). Числа — только из источника (правило 4), цены — с «уточняйте» + дата. Валюта — **македонский денар (MKD)**; курс де-факто привязан к евро (~61,5 MKD/EUR), поэтому удобно давать ориентир в EUR в скобках.
5. **2–4 фото** (фото обязательны: ≥3 инлайн-figure ИЛИ ≥6 gallery — DoD) с осмысленными `alt` и подписями (figcaption) — см. §6.
6. **Блок внутренних ссылок** на смежные статьи — минимум 2 (см. §7), плюс хаб раздела.
7. **Дата обновления** — поле `updatedAt` (видно на странице). Поднимать при правке фактов.
8. **Объём 1200–2000 слов, без «воды».** Каждый абзац несёт пользу; никакой генерёнки и near-duplicate (§15 — риск бана).
9. **Доп. требования проекта:** `description` ≤155 с запросом; факты не выдумывать (правило 4) — нет данных → TODO/«уточняйте» + источник в `sources`; уникальная фактура (≥3 своих факта), не перепечатка; **FAQ не используем** (убран со всего сайта — новую инфо вплетать в тело). Для «денежных» гайдов — `yearInTitle: true` (год в H1/title).

## 9. Проверка перед публикацией (гейты)

Из корня проекта:

| Команда | Что проверяет |
|---|---|
| `npm run check` | типы + соответствие frontmatter схеме |
| `npm test` | enums · **parity (валидация en-контента: `lang` ↔ папка, title ≤60)** · photos (cover ≤200 КБ) · **interlinks (≥2)** |
| `npm run test:links` | внутренние ссылки резолвятся + `/go/`-партнёры валидны |
| `npm run lint` | стиль/ESLint |
| **`npm run qa`** | **всё вышеперечисленное + чистая сборка + аудит `dist/` → вердикт GO/NO-GO** |

**Правило:** перед коммитом — `npm run qa` = **GO**. Затем коммит en-версии + push в `main`.

**Как убедиться, что статья на месте с корректным hreflang:**
1. `npm run build` → проверь, что появился `dist/<category>/<slug>/index.html`.
2. В нём есть `<link rel="alternate" hreflang="en" …>` + `<link rel="alternate" hreflang="x-default" …>` (оба на canonical URL) — проставляется автоматически (сайт одноязычный).
3. `npm run test:links` — без битых ссылок.

## 12. Как добавлять ЗАВЕДЕНИЕ (ресторан / услугу) — это делаю я

Заведения — это НЕ лонгрид-статьи, а карточки-директории:
- **Ресторан / кафе / бар / винотека** → коллекция `restaurants/` → раздел «Где поесть» `/food/`.
- **Услуга** (риелтор / клининг / переезд / ремонт / документы / переводчики) → коллекция `services/` → `/relocation/services/`.

### Жёсткие правила отбора (обязательно)
1. **Только реальные заведения** с проверяемыми данными. Вымышленные/«примеры» в директорию проверенных мест НЕ вносим (правило 4, §15).
2. **Консенсус > 4★ из НЕСКОЛЬКИХ источников** + наш `ratingNote`/`review` своими словами (чужие отзывы дословно нельзя, §15).
3. **Фото обязательно** (`cover`; для CC — `coverCredit`). Цены/часы (`dishes`/`hours`) — только из источника, иначе не указывать. Цены — в денарах (MKD).
4. **Только en-версия** (сайт одноязычный).
5. **🔵 Метка «Проверено» — ОБЯЗАТЕЛЬНА** (правило владельца, §13 ниже).

### Шаги
1. Скаффолд: `npm run new -- restaurant <slug> --city "Скопье"` (или `service <slug>`) → создаст только en + папку фото.
2. Заполни обязательные поля. **Ресторан:** `name`, `city`, `priceLevel` (`$`/`$$`/`$$$`), `cuisineKey` (из enum: `macedonian`/`balkan`/`seafood`/`wine`/`cafe`/`bakery`/`vegetarian`/`grill`/`street`/`bar`), `geo.coord`, `mapUrl`, `ratingNote`, `sources`, `summary`, **`verifiedAt`**, `cover`. **Услуга:** `name`, `rubric` (из enum: `housing-realtors`/`cleaning`/`moving`/`repairs`/`documents`/`translators`), `summary`, `city`, реальный `url`/`contact` (без example.com), **`verifiedAt`**.
3. Фото (§6); `verifiedAt` = дата нашей **реальной** сверки данных.
4. Снять `draft` → `npm run qa` = GO → коммит.

### Добавить заведение в витрину на главной (с флагом города)
В `HomePage.astro` → `showcasePicks` добавить pick с полем `city` (`city: 'Скопье'`) → на карточке появится пин-чип «📍 Город» (см. память `home-showcase-monetization`). В витрину помещаем только заведения с меткой «Проверено» (§13).

## 13. 🔵 Правило: метка «Проверено» обязательна у КАЖДОГО заведения/услуги

**КАЖДОЕ заведение и услуга, которые мы вносим, обязаны нести видимую метку «Проверено» (галочка ✓)** — это обещание директории «проверенные места» и ключевой сигнал доверия (E-E-A-T).

- **Чем:** единый компонент **`VerifiedBadge`** (галочка ✓ + «Проверено · {дата}» + расшифровка «опытными путешественниками, гидами и местными»). Дата — из поля `verifiedAt`. **Не изобретать новых галочек** — один знак доверия на весь сайт (консистентность).
- **Честность (правило 4):** `verifiedAt` ставим ТОЛЬКО при реальной сверке (данные сверены с источниками/на месте). «Проверено» на непроверенном — запрещено.
- **Текущее состояние:** рестораны (`/food/`) — `verifiedAt` обязателен схемой → полный бейдж «Проверено · {дата}» ✓. Услуги (`/relocation/services/`) — поле `verifiedAt` в схеме `services` опционально + **компактный** `VerifiedBadge` (проп `compact`: ✓ + «Проверено», без даты) в карточках; ставить только реально сверенным услугам (демо/вымышленные — без, с бейджем «Пример» + noindex).
- **Компонент:** `VerifiedBadge` имеет `compact` (для плотных карточек — ✓ + «Проверено») и полный (✓ + «Проверено · {дата}» + расшифровка — для страницы места/шапки статьи).
- **Метка «Проверено» — БЕЗ даты у заведений** (решение владельца; `verifiedNoDate: true`): `RestaurantCard` рендерит компактный бейдж «Проверено» без даты. `verifiedAt` остаётся обязательным в схеме `restaurants` (хранится для записи), но в карточке может не показываться.
- **Знак качества** — золотая медаль `/images/quality/seal-gold.webp` (выбор владельца; запасные SVG-печати `seal-star`/`seal-rosette`/`seal-shield`/`seal-laurel`/`seal-check` там же).
  - **ОПТ-ИН, только по запросу владельца.** Поле `qualityMark: true` у ресторанов И у статей (`articleBase`); по умолчанию выкл. **Перед созданием любого заведения — СПРОСИТЬ владельца, нужен ли знак.**
  - **Размер ФИКСИРОВАН и един:** карточки (витрина `.scard__quality`, `/food/` `.rcard__quality`, миниатюра хаба `.card__quality`) — **62px**, выходит за правый верхний угол + золотое свечение; обложка-герой статьи (`.cover__quality`) — 72px, в углу. Не делать адаптивным.
  - Сменить знак везде = поменять путь в `ShowcaseRail.astro` (`QUALITY_MARK`), `RestaurantCard.astro`/`ArticleCard.astro`/`ArticlePage.astro` (`*__quality` src). Подробности — память `quality-mark-establishments`.
- **Сайт + Instagram заведения** (правило владельца): **отдельные поля** `website` (личный сайт → кнопка «Сайт», `globe`) и `instagram` (→ кнопка «Instagram», icon `instagram`) — у ресторанов и статей-мест. Рендерятся в карточке `/food/`, поповере витрины и статье. **При добавлении ЛЮБОГО заведения СПРАШИВАТЬ у владельца Instagram** (память `establishment-instagram`). **Личный сайт НЕ убирать** — показываем обе кнопки, если есть и сайт, и IG.
- **Фото заведений: карусель + лайтбокс.** Несколько фото = `cover` + `gallery[]`; в карточке листаются (стрелки на наведении/свайп, `/js/rcard-gallery.js`), клик по фото → полноэкранный просмотр с листанием (`/js/lightbox.js`).
- **Витрина на главной поддерживает заведения:** в `HomePage.astro` → `showcasePicks` добавить `{ kind: 'restaurant', slug: '<slug>', edaSlug: 'where-to-eat-<город>', kicker: 'food' }` → карточка с обложкой заведения, ссылкой на страницу города в «Еде» и локализованным пин-чипом города (берётся из `city` карточки). В витрину — только заведения с меткой «Проверено».

## 14. Факты и источники (YMYL и обычные)

- **Обычные факты** (цены, расстояния, часы, гео): из надёжного источника, домен/ссылку — в `sources`; цены — с «уточняйте» + датой. Нет данных → `<!-- TODO: факт от владельца -->`, не выдумывать (правило 4).
- **YMYL-факты (визы / налоги / ВНЖ / бизнес) — особый режим (правило 5).** Только из официальных первоисточников + видимый дисклеймер «уточняйте, правила меняются». Первоисточники: **mfa.gov.mk** (визы), **ujp.gov.mk** (налоговая), **customs.gov.mk** (таможня/въезд), **stat.gov.mk** (статистика), **PwC Tax Summaries** (налоги). Брифы YMYL-фактов держим в `briefs/_SOURCES-YMYL.md`. Проверенные опорные факты (с дисклеймером «сверяйте с официальными источниками»):
  - **Визы:** US/UK/EU/Украина — безвиз **90/180**; **граждане РФ — нужна виза** (с 21.03.2022; исключение — держатели действующих виз US/UK/CA/Шенген C: безвиз 15 дней). Регистрация в полиции в течение 48 ч (обычно делает отель).
  - **Цифровой номад-визы НЕТ** — «nomad visa Macedonia» в SEO-агрегаторах выдумка; долгое пребывание — ВНЖ через работу/бизнес.
  - **Налоги:** **10% плоский НДФЛ**, **10% налог на прибыль (CIT)**, **18% НДС** (льготные 5%/10%), соцвзносы 28%. Бизнес-форма — DOO/DOOEL (допускается единственный учредитель).
  - **Валюта:** македонский денар (MKD), де-факто привязан к евро — паритет ~**61,5 MKD/EUR**.
- **Не повторять опровергнутые мифы:** «безвиз для граждан РФ» и «номад-виза Македонии» (правило 5).

## 15. Антибан / качество (коротко)

- **Без SEO-генерёнки.** Каждая статья — уникальная фактура (≥3 своих факта), не перепечатка; near-duplicate и массовые тонкие страницы = риск бана (SPEC §15).
- **Чужие отзывы/рейтинги дословно не копируем** (ToS Google/TripAdvisor) — используем как критерий отбора + ссылку, добавляя своё описание (`ratingNote`/`review`).
- **Фото — только свои или CC/PD** с верной атрибуцией; не выдавать чужую страну/валюту за македонскую.
- **Перф-бюджет (мобайл):** LCP < 2.0 с, CLS < 0.1, JS на странице < 50 КБ; webp ≤200 КБ.
- **YMYL — только официальные первоисточники** + дисклеймер (§14, правило 5).
